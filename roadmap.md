## Data

Data is the first bottleneck of large language model training. A model does not learn "the internet" directly; it learns from a carefully constructed training mixture. The data pipeline decides what knowledge, reasoning style, domain coverage, language distribution, and failure modes the model will inherit.

### Web data

The largest source for pretraining is web data. Web data is broad and cheap, but it is messy. It contains HTML pages, PDFs, code repositories, forums, papers, books, tables, boilerplate, advertisements, duplicated pages, machine-translated text, spam, and generated content.

The basic pipeline is:

1. source data
2. extract text and metadata
3. clean and normalize
4. deduplicate
5. filter low-quality or unsafe content
6. classify by domain and language
7. design the final data mixture
8. tokenize and pack examples for training

The first task is large-scale crawling or scraping. The crawler collects raw documents from the web, stores URLs and metadata, and records enough provenance to support later filtering. The extraction step then converts different formats into usable text. This is nontrivial because each source has different structure:

- webpages contain navigation bars, comments, ads, and hidden text
- PDFs may have broken reading order, equations, tables, and footnotes
- books and papers may need section structure and citation cleanup
- code repositories mix code, comments, documentation, logs, and generated files
- forums and Q&A sites contain conversations rather than independent documents

After extraction, the data must be cleaned. Cleaning removes boilerplate, broken encodings, repeated headers, navigation text, excessive whitespace, malformed text, and documents with too little useful content. Language identification and domain classification are usually applied here.

Deduplication is essential. If the same page appears many times, the model may memorize it and the effective dataset size becomes much smaller than expected. Deduplication is usually done at multiple levels:

- exact document deduplication
- near-document deduplication
- paragraph-level deduplication
- benchmark contamination detection

Filtering removes low-quality documents. Simple filters use document length, character distribution, language confidence, punctuation ratio, repetition ratio, toxicity scores, or perplexity from a reference model. More advanced filters use classifiers trained to distinguish high-quality educational, technical, or reference-style text from spam and low-value content.

The final step is mixture design. A pretraining corpus is not just "all collected data." Different sources are sampled with different weights. For example, code, math, papers, books, general web pages, multilingual data, and instruction-like text may all be mixed in different proportions. Mixture design affects the final model strongly: more code improves coding and structured reasoning, more math improves mathematical fluency, more multilingual data improves language coverage, and more high-quality long-form text improves coherence.

### Synthetic data

High-quality web data is finite, and much of the useful public web has already been used by modern models. Synthetic data is therefore increasingly important. Synthetic data is generated by models, tools, templates, agents, or simulators rather than scraped directly from the web.

Synthetic data is useful when we need targeted coverage. For example, we can generate:

- math problems with verified solutions
- theorem explanation and proof-writing examples
- coding tasks with tests
- tool-use trajectories
- question-answer pairs from documents
- domain-specific instruction data
- reasoning traces for multi-step problems
- counterexamples and adversarial questions

A common pattern is generation plus verification. One model or agent proposes a problem and solution, and another verifier checks the answer. For math, the verifier may be symbolic computation, unit tests, formal proof checking, numerical checking, or a stronger model. For code, the verifier can be test execution. For retrieval tasks, the verifier can check whether the answer is supported by the source document. For factual tasks, generation should usually be followed by a check-and-rewrite loop. The model first drafts an answer, then searches for supporting sources, checks whether each important factual claim has evidence, compares multiple sources when possible, and rewrites unsupported or inconsistent claims. This reduces hallucination because the final example is not accepted only because it sounds plausible; it must be grounded in retrieved evidence.

Synthetic data can also be generated from templates. Templates are usually written by domain experts and used as seeds to generate many related examples. They are useful when the structure of the task is clear, such as algebra transformations, theorem recall, concept comparison, or tool-call formatting. Templates can also be meta-templates: instead of directly producing one example, they instruct an AI system to generate more problems, answers, variations, or verification cases.

One important use of synthetic data is to expand existing high-quality text. Given a textbook section, paper, documentation page, or lecture note, an AI system can rewrite it more clearly, add missing intermediate steps, create examples, generate exercises, and produce question-answer pairs. This turns one source document into many training examples while keeping the examples grounded in real material.

For example, from a theorem statement, we can generate:

- a plain-language explanation of the theorem
- a proof sketch and a detailed proof
- examples where the theorem applies
- counterexamples showing why each assumption is needed
- questions asking for related definitions and concepts
- comparison questions between this theorem and nearby results
- exercises that ask the model to use the theorem in a new setting

This kind of synthetic expansion is especially useful for math, science, coding, and technical domains. The base text provides the factual anchor, while the generation process creates diverse supervision formats: explanations, questions, answers, examples, exercises, and concept maps. The generated data should still be checked, because the model may add incorrect details when it tries to fill gaps in the original text.

Agents make synthetic data more flexible. An agent can search, write, solve, critique, revise, and verify. This makes it possible to build complex training examples such as research workflows, theorem exploration, multi-hop question answering, and tool-use traces.

### Data for different training stages

Different stages need different data.

Pretraining needs massive general data. The data should cover a wide range of subjects, including everyday language, science, mathematics, engineering, medicine, law, economics, history, philosophy, social science, literature, education, programming, documentation, forums, news, and multilingual text. Pretraining data is usually measured in trillions of tokens, and even small mixture choices can strongly affect the final model. Synthetic data can be added at massive scale to improve coverage in weak areas, such as math reasoning, code tasks, rare languages, domain-specific explanations, or carefully verified question-answer pairs.

Supervised finetuning needs instruction-response data. The goal is to adapt the answering style of pretrained model to everyday use cases. SFT teaches the model how to collaborate with the user, not just predict the next token. The data should show how to clarify ambiguous requests, preserve user intent, ask for missing information when necessary, follow formatting constraints, avoid unsupported claims, and give answers at the right level of detail. The mixture usually includes single-turn instructions, multi-turn conversations, demonstrations of tool use, retrieval-augmented answers, safe refusal examples, coding tasks, math solutions, writing tasks, and domain-specific workflows. Some SFT seed data can also come from the RL phase: prompts that expose model failures, high-reward answers, corrected low-reward answers, verifier-approved solutions, and preference-labeled examples can be converted back into supervised examples for another round of finetuning.

Reinforcement learning needs prompts, candidate answers, preference labels, reward signals, and verifiers. The data mixture includes real user prompts, synthetic prompts, difficult math and coding problems, tool-use tasks, retrieval tasks, safety-sensitive prompts, preference comparisons, rejected answers, and verifier-scored rollouts. For tasks with objective answers, such as math, code, formal reasoning, retrieval, and tool calling, the reward can come from tests, proof checkers, execution results, source matching, or rule-based verifiers. For subjective tasks, such as helpfulness, writing quality, tone, and preference, the reward usually comes from human comparison labels or a reward model trained on those labels. Synthetic data and RL are closely connected because RL can create new data through rollout: verified solutions, search-grounded explanations, tool-use traces, and reasoning trajectories. These can be folded back into the corpus to improve the next model.

In short, data work is not a preprocessing detail. It is one of the main design problems in LLM training. A strong data pipeline combines web-scale collection, careful filtering, mixture design, synthetic generation, and verification. We will discuss this in [data.md](./sections/data.md)

## Pre-training


### Data diagnosis

Data diagnosis asks where model errors come from. If the model hallucinates, repeats misinformation, fails on a domain, or learns an unwanted style, we need to trace the behavior back to data shards, source types, or filtering choices. Bad shards can then be rewritten, downweighted, removed, or replaced with better data.

In practice, this requires keeping provenance metadata for each document, such as source, crawl time, language, domain label, filter scores, and shard id. Without provenance, it is hard to connect an observed model error to the data that caused it.

### Data quality and data mixture

Data quality decides which documents are worth training on. Data mixture decides how much weight each source receives. The mixture must balance domains, languages, code, math, synthetic data, high-quality books, papers, and web text. This choice strongly affects final capability.

In practice, mixture design should be treated as an experiment. Train smaller models or short runs with different mixtures, evaluate them on target benchmarks, then use the result to decide the large run.

### Contamination control

Contamination control removes benchmark leakage and duplicated evaluation examples. If test examples appear in training data, evaluation scores become unreliable. This is especially important for popular math, coding, and QA benchmarks that are widely copied online.

In practice, exact matching is not enough. Near-duplicate detection, paraphrase detection, URL/source filtering, and manual checks are often needed for important evaluation sets.

### Training phases

Training data does not need to be shown in one fixed mixture from start to finish. A common strategy is to train first on broad web-scale data, then increase the weight of higher-quality, synthetic, or domain-specific data later. The order of data can affect stability, final knowledge, and specialization.

In practice, phase boundaries should be planned before training starts, but still monitored during training. If loss, evaluation scores, or stability changes sharply after a data switch, the schedule may need to be adjusted.

### Stability

Pretraining is sensitive to data bugs and optimization settings. We need to monitor loss spikes, gradient norms, optimizer behavior, numerical precision, tokenization problems, and corrupted batches. Stability problems can come from either the model/training setup or from bad data.

In practice, training should log enough information to debug failures: batch ids, data shards, optimizer state, hardware errors, skipped batches, and checkpoint health. Frequent checkpoints are necessary because a large run can fail after consuming substantial compute.

### Architecture design and parameter tuning

Pretraining also depends on model and training choices: model size, context length, tokenizer, learning rate, batch size, regularization, optimizer, and parallel training strategy. These choices interact with the data mixture, so tuning them cannot be separated completely from data design.

In practice, these choices are constrained by hardware. Memory, communication bandwidth, sequence length, and checkpoint cost often determine which architecture or batch size is actually feasible.

### Continued pretraining or domain adaptation

Continued pretraining adapts a base model to a target domain such as math, code, science, medicine, or law. The key problem is to improve the target capability without destroying general ability. This requires a careful domain mixture, learning rate, training length, and evaluation plan.

In practice, domain adaptation should include both domain benchmarks and general benchmarks. If only domain scores are monitored, the model may silently lose general instruction-following, language quality, or safety behavior.

We will discuss this in [llm.md](./sections/llm.md). 
 

## Supervised finetuning

### Instruction data mixture

Supervised finetuning adapts a pretrained model to user-facing behavior. The data mixture should include single-turn instructions, multi-turn conversations, writing tasks, summarization, coding help, math solutions, tool-use examples, retrieval-augmented answers, safety examples, and domain-specific workflows.

In practice, the mixture should reflect the expected product use cases. A model for coding needs many code-editing and debugging conversations, while a model for education needs explanations, examples, misconceptions, and step-by-step tutoring.

### User collaboration

SFT should teach the model to collaborate with the user, not only produce a final answer. The model should learn to preserve user intent, ask clarifying questions when necessary, follow formatting constraints, adapt to the user's level, and avoid doing unnecessary work.

In practice, multi-turn data is important because many real tasks are interactive. The data should include correction, refinement, partial information, follow-up questions, and cases where the model must update its answer after the user gives new constraints.

### Tool use and retrieval

Many useful assistants need tools: search, code execution, file editing, calculators, databases, or APIs. SFT data should show when to use a tool, how to call it, how to interpret results, and how to explain the final answer.

In practice, tool-use data should include failed tool calls, missing information, conflicting search results, and source-grounded answers. This prevents the model from treating tool output as magic and helps it learn robust workflows.

### Safety and refusal behavior

SFT also teaches boundaries. The model should learn when to answer directly, when to refuse, when to redirect to safer information, and when to avoid unsupported factual claims.

In practice, safety data should not only contain refusals. It should also include allowed neighboring requests, safe completions, and examples where the model gives useful help without crossing the boundary.

### Reasoning and chain-of-thought data

Reasoning data can help on math, coding, planning, and technical explanation. However, noisy reasoning traces are harmful because the model may learn to imitate plausible but incorrect reasoning.

In practice, reasoning data should be verified. For many user-facing tasks, a concise explanation with the important steps is better than a long chain of thought. For math and code, final answers should be checked by tests, symbolic tools, or independent verification when possible.

### Evaluation and iteration

SFT should be evaluated on the actual behaviors it is supposed to teach: instruction following, helpfulness, factuality, formatting, tool use, coding, math, safety, and multi-turn consistency.

In practice, SFT is usually iterative. Failed prompts, corrected answers, high-quality user conversations, verifier-approved outputs, and examples from the RL phase can be converted into new supervised data for the next round.
